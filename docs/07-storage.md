# TimePro — Storage Strategy

> **Implementation status** — ✅ built · ⛔ planned.
>
> - ✅ MVP storage is the **local filesystem**: screenshots are written to `STORAGE_DIR` (default `apps/api/data/screenshots/{org}/{date}/{id}.png`) and served via `GET /v1/screenshots/:id/raw`.
> - ⛔ Not yet built: S3/object storage, thumbnail generation, envelope encryption (KMS), lifecycle/retention tiering, signed URLs, and CDN. The S3 key layout below is the target.

Object storage is the dominant cost driver. The design optimizes for: (1) cheap writes from agents, (2) cheap reads of thumbnails in the gallery, (3) cold storage for older fulls, (4) clean per-tenant deletion.

## 1. Buckets

| Bucket                              | Region        | Purpose                              | Class                |
| ----------------------------------- | ------------- | ------------------------------------ | -------------------- |
| `timepro-screenshots`             | Primary       | Original + thumbnail webp            | Lifecycle-tiered     |
| `timepro-exports`                 | Primary       | CSV/XLSX/ZIP exports                 | Standard, 7-day TTL  |
| `timepro-agent-updates`           | Primary       | Tauri update bundles + signatures    | Standard             |
| `timepro-archive`                 | Glacier IR    | Cold time entry archives (Parquet)   | Glacier Instant      |
| `timepro-backups`                 | Cross-region  | Postgres base backups + WAL          | Standard-IA          |

All buckets: object lock not enabled by default; offered per-org for compliance plans.

## 2. Key Layout

### 2.1 Screenshots

```
s3://timepro-screenshots/
  org/{org_id}/
    {yyyy}/{mm}/{dd}/
      user/{user_id}/
        full/{screenshot_id}.webp
        thumb/{screenshot_id}.webp
```

Reasons:
- Org-prefixed: lifecycle rules, per-tenant export, "delete an org" is `aws s3 rm --recursive`.
- Date-partitioned: lifecycle transitions and analytics queries are cheap.
- `full/` and `thumb/` siblings: separate lifecycle rules for the two classes.

### 2.2 Exports

```
s3://timepro-exports/
  org/{org_id}/{export_id}.{csv|xlsx|zip}
```

Lifecycle: delete after 7 days (S3 expiration). Re-export if needed.

### 2.3 Agent updates

```
s3://timepro-agent-updates/
  {channel}/{platform}/{arch}/
    {version}/
      app.{ext}                 # dmg / msi / appimage / deb
      app.{ext}.sig             # ed25519 signature
    latest.json                 # manifest with current version + sig
```

Served via CloudFront with a short TTL on `latest.json` (60s) and long TTL on versioned bundles (immutable, 1y).

## 3. Encryption

Envelope encryption.

- **KEK** (Key Encryption Key) per org, held in AWS KMS (key alias `alias/timepro/org/<id>` — actually a single CMK with per-org context for cost; full per-org CMKs offered in Enterprise plan).
- **DEK** (Data Encryption Key) per screenshot, 256-bit, generated by `GenerateDataKey`.
- The ciphertext blob in S3 is `[dek_ciphertext (256b) || nonce (12b) || AES-256-GCM(payload)]`.
- For thumbnails, we use the same DEK (rolled into the row in `screenshots.encryption_dek_id` linking to a key cache row).
- KMS calls are cached per-org for 5 minutes (the GenerateDataKey result is reused within a small window) to control KMS spend.

Server-side: S3 SSE-KMS is *also* enabled as a backstop. So objects are double-encrypted: app-layer (our DEK, owned by us) on top of S3 SSE-KMS (managed by AWS). Loss of KMS = unreadable.

## 4. Thumbnail Pipeline

We push thumbnail generation to the agent when possible (saves API bandwidth + server CPU). Server-side fallback exists.

- Agent encodes a 320×180 webp at q=75 alongside the full image; both are uploaded.
- If the agent didn't produce a thumb (e.g., older version), the `screenshot.process` worker generates it via `sharp` from the decrypted full.

Thumbs are always served from CDN; fulls are signed-URL direct from S3 (with short TTL, no CDN — we don't want CDN caching of sensitive images keyed by anything other than the signed URL).

## 5. Lifecycle Rules

| Class    | Prefix                                | Day 0   | Day 30   | Day 90        | Day 365       | Day 2555 (7y) |
| -------- | ------------------------------------- | ------- | -------- | ------------- | ------------- | ------------- |
| Full     | `org/*/yyyy/mm/dd/user/*/full/*`      | Standard| IA       | Glacier IR    | Glacier Deep  | Delete        |
| Thumb    | `org/*/yyyy/mm/dd/user/*/thumb/*`     | Standard| Standard | Standard      | IA            | Delete        |
| Exports  | `org/*/*.{csv,xlsx,zip}`              | Standard| —        | —             | —             | Delete (7d)   |

Per-org overrides:
- "Retain forever" plans skip the Day-2555 expiration.
- "Privacy mode" plans expire fulls on Day 30 (thumbnails optional).
- Per-user retention is allowed but bounded by org policy.

Lifecycle is configured per bucket; the retention worker also emits explicit deletes to handle non-uniform per-org policies (S3 lifecycle is one rule per bucket, so we use tags + tag-based lifecycle):

```
PutObjectTagging:
  org_id = <uuid>
  retention_class = standard | extended | privacy
```

Three lifecycle rules — one per `retention_class` tag.

## 6. Signed URLs

| Surface                  | Mechanism                | TTL  | Notes                              |
| ------------------------ | ------------------------ | ---- | ---------------------------------- |
| Screenshot view (single) | S3 SigV4 pre-signed URL  | 60s  | Per-request                        |
| Gallery browsing         | CloudFront signed cookie | 5min | Scoped to `/org/<id>/.../thumb/*`  |
| Agent upload             | S3 SigV4 pre-signed PUT  | 5min | Plus required headers              |
| Export download          | S3 SigV4 pre-signed URL  | 60min| Sent in email                      |

Audit log entry on every URL issuance carries `(user_id, screenshot_id, purpose)`.

## 7. CDN

CloudFront in front of:
- `timepro-screenshots/org/*/thumb/*` — signed cookies, 5-min TTL.
- `timepro-agent-updates/*` — public, immutable for versioned paths.
- `web` static assets (`/_next/static/*`) — long cache.

Web app (HTML / RSC) uses CloudFront pass-through with no caching (origin is `web` service).

Reasons fulls do not go through CDN:
- They're sensitive — single-use signed URLs are easier to audit if direct-from-S3.
- They aren't viewed often (admins, occasional review) — CDN warm-up has limited benefit.
- Avoids needing a custom origin function for per-request access control.

## 8. Retention & Deletion

> ✅ **Built (interim, local-disk):** the org-wide `screenshots.retention_days` setting (1/3/6/12 months or Forever;
> default 3 months) drives `apps/api/src/lib/retention.ts`, which hard-deletes old screenshot rows **and unlinks the
> local files**. With no scheduler/worker yet (Phase 8), the sweep runs **in-process** from `server.ts` (~30s after
> boot, then every 12h); `POST /v1/admin/screenshots/prune` triggers it on demand. The target design below
> (soft-delete window + S3 lifecycle tiers + a `retention.sweep` worker) supersedes this once object storage lands.

Target design (two layers):

1. **Logical**: `screenshots.status = 'deleted'`, hidden from UI immediately, kept for 30-day soft-delete window.
2. **Physical**: `retention.sweep` worker deletes the underlying S3 objects + DB rows after the soft-delete window or lifecycle expiry.

When an org is deleted:
- Immediate: org marked `status='cancelled'`, all API access blocked, agents revoked.
- 30-day grace: data still in place, can be restored.
- Day 30: `org.purge` job runs — deletes S3 prefix (`aws s3 rm --recursive`), drops all tenant rows (RLS-bound), removes KMS key (or schedules deletion if dedicated CMK).
- Final tombstone: `audit_logs.platform_*` record stays 7 years.

GDPR DSAR:
- "Export my data": user-scoped export job aggregates time entries, activity, screenshots metadata, app/url logs into a ZIP.
- "Delete my data": user soft-deleted; tied data anonymized (`user_id` retained for foreign-key integrity, PII zeroed) or deleted depending on org policy and legal hold flags.

## 9. Cost Optimization

Per-screenshot cost components (rough order):

| Component       | Cost driver                | Mitigation                                              |
| --------------- | -------------------------- | ------------------------------------------------------- |
| S3 PUT          | Number of uploads          | Compress agent-side; batch thumbs in same PUT (multipart)|
| S3 storage      | GB-months                  | Tiering, WebP encoding (4–6× smaller than PNG)          |
| Egress          | Bytes served               | Thumbnails default in UI; signed URL for full only on click |
| KMS GenerateKey | KMS calls                  | Cache DEK for 5 min per org                             |
| CloudFront      | GB delivered + requests    | Long TTL on versioned assets; no CDN on fulls           |

Sizing assumptions:
- 60 screenshots/hour × 8h/day × 22 working days = ~10,560 per user/month.
- WebP at quality 70 typical 1920×1080 ≈ 80 KB; thumb ≈ 12 KB.
- Per user/month storage: ~960 MB fulls + 130 MB thumbs ≈ 1.1 GB.
- 100 orgs × 50 users = 5000 users × 1.1 GB = ~5.5 TB/month new storage. After tiering (90d to Glacier), steady-state ≈ 16 TB.
- See [10-scaling.md](10-scaling.md) for full projections at 1k / 10k orgs.

## 10. Multi-Region (Phase 3)

- Per-org `data_region` column. The agent uses that to pick its bucket and API endpoint.
- Cross-region replication is **not** the default — it doubles cost and creates compliance complexity.
- Replication is opt-in for enterprise customers (e.g., for warm DR in another region of the same legal jurisdiction).
- Backups always cross-region (different bucket).
